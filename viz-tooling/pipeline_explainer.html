1. Load config

The YAML file is merged with defaults, validated, and converted into a Config object. This decides schema mode, model roles, taxonomy depth, target size, concurrency, and evaluation behavior.

Model Roles

The roles are job labels. They can point to the same model in YAML, but the code uses them for different kinds of thinking.

strategic · bulk · critic
strategic

The planner

Designs the coverage space before generation starts.

Discovers top-level factors like issue type, urgency, tone.
Expands taxonomy nodes into child categories.
Refines child lists to reduce duplication.
Creates sampling strategies from the taxonomy.
bulk

The producer

Turns sampled requirements into actual synthetic records.

Creates meta-prompts from a taxonomy mix.
Optionally complexifies a meta-prompt.
Generates JSON records or free-text outputs.
Repairs invalid JSON and refines rejected rows.
critic

The judge

Checks quality and optional evaluation judgments.

Accepts or rejects a generated row semantically.
Explains why a row failed.
Optionally reassigns rows to taxonomy nodes.
Optionally scores relative complexity.

Taxonomy Building

A taxonomy is a set of trees. Each tree is one factor, meaning one axis of variation the dataset should cover.

tree per factor

issue_type

The problem being reported. Click nodes in the tree to see the JSON shape that gets stored.

What breadth-first expansion means
First the model expands every root into level-1 children. Only after that does it expand those level-1 children into level-2 children. This keeps the taxonomy balanced instead of overdeveloping one branch early.
What best_of_n means
For each node, the code can ask for several child proposals, combine them, then ask the model to refine the list. This gives the taxonomy critic more raw ideas to choose from.

Selected node JSON

Taxonomy nodes are plain JSON objects with name, description, level, path, and children.

Strategy vs Taxonomy Mix

The strategy is the reusable sampling recipe. The taxonomy mix is the exact sampled combination for one generated row.

Config Knobs And Current Limits

Some behaviors are user-configurable today. A few reasonable knobs, like dedupe n-gram size and threshold, are currently hardcoded.

what users can tune

Configurable now

taxonomy.depth: how many levels each factor tree gets.
taxonomy.best_of_n: how many raw child proposals are generated before refinement.
taxonomy.children_per_node: requested child count per expansion.
generation.complexity_ratio: probability that a row's meta-prompt gets complexified.
generation.overgenerate_ratio: how many extra attempts to make before final trimming.
evaluation.dedupe: whether dedupe runs.

Hardcoded today

Dedupe uses 13-gram overlap by default.
Dedupe threshold defaults to 0.8 Jaccard similarity.
JSON repair currently gets one tightly scoped repair attempt.
Coverage-aware trimming greedily maximizes new taxonomy paths.
# Possible future config shape:
evaluation:
  dedupe: true
  dedupe_ngram_size: 13
  dedupe_threshold: 0.8
